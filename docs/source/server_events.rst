服务器事件
==================

我们知道，在服务端与客户端的连接期间，为了取得必要的信息，客户端往往是主动发起通信的角色。
然而，在某些情况下，例如服务器的防范锁定模式被自动或人为地激活，
服务端的状态发生了会影响某个乃至某一部分客户端运作的变化——为了通知客户端为这一变化做好准备，
我们便会发现服务端的主动通信是必要的。

无疑，服务端不会无缘无故地向客户端发起通信：如果真的发生了需要服务端主动发起通信的情况，
我们便可以断定其背后一定存在某个 **事件** 作为它的原因。可以认为，服务端向客户端发起通信，
便是在向客户端传递关于这个事件的详情。

我们定义服务器向客户端主动发起的通信中携带的信息为一个 **服务器事件**。服务端能且仅能在 
**一次** 主动通信中用 **一个** 消息帧来传递一个服务器事件。
这意味着这样的一次通信中应有且仅有一个类型为 ``FrameType.CONCLUSION`` 的消息帧，
且不允许存在任何其他类型的消息帧。

通信规范
------------

发送
^^^^^^^^

服务器事件应当以如下的格式发送给客户端：

.. code-block:: json

   {
        "type": "object",
        "properties": {
            "event": {"type": "string"},
            "data": {"type": "object"},
        },
        "required": ["event", "data"],
    }

其中， ``event`` 是服务器事件的名称，而 ``data`` 是一个对象，包含了与该事件相关的数据。

响应
^^^^^^^^

客户端不应对传递服务器事件的通信进行任何响应。

事件
--------------

服务器事件的 ``data`` 的结构根据事件的不同而有所不同。

下列表格的标题为事件名称，表格内容为该事件的 ``data`` 的结构。

.. list-table:: lockdown
    :header-rows: 1

    * - 字段
      - 类型
      - 取值
      - 说明
    * - status
      - 必需
      - ``true``, ``false``
      - 取值为 ``true`` 时表示服务器处于防范锁定模式。
    * - reason
      - 可选
      - string
      - 仅当服务器处于防范锁定模式时被提供。指示防范锁定模式被激活的原因。
